## Beads Issue Tracker

This project uses **bd (beads)** for issue tracking. Run `bd prime` to see full workflow context and commands.

### Quick Reference

```bash
bd ready              # Find available work
bd show <id>          # View issue details
bd update <id> --claim  # Claim work
bd close <id>         # Complete work
```

### Rules

- Use `bd` for ALL task tracking — do NOT use TodoWrite, TaskCreate, or markdown TODO lists
- Run `bd prime` for detailed command reference and session close protocol
- Use `bd remember` for persistent knowledge — do NOT use MEMORY.md files

**Architecture in one line:** issues live in a local Dolt DB; sync uses `refs/dolt/data` on your git remote; `.beads/issues.jsonl` is a passive export. See https://github.com/gastownhall/beads/blob/main/docs/SYNC_CONCEPTS.md for details and anti-patterns.

## Session Completion

**When ending a work session**, you MUST complete ALL steps below. Work is NOT complete until `git push` succeeds.

**MANDATORY WORKFLOW:**

1. **File issues for remaining work** - Create issues for anything that needs follow-up
2. **Run quality gates** (if code changed) - Tests, linters, builds
3. **Update issue status** - Close finished work, update in-progress items
4. **PUSH TO REMOTE** - This is MANDATORY:
   ```bash
   git pull --rebase
   git push
   git status  # MUST show "up to date with origin"
   ```
5. **Clean up** - Clear stashes, prune remote branches
6. **Verify** - All changes committed AND pushed
7. **Hand off** - Provide context for next session

**CRITICAL RULES:**
- Work is NOT complete until `git push` succeeds
- NEVER stop before pushing - that leaves work stranded locally
- NEVER say "ready to push when you are" - YOU must push
- If push fails, resolve and retry until it succeeds

## Tech Stack

Go 1.23 · Gin · PostgreSQL 16 (`pgx/v5`) · Redis 7 (`go-redis/v9`) · JWT HS256 (access)
+ opaque refresh tokens · `golang-migrate` (SQL migrations) · `go-playground/validator` ·
SMTP/gomail (email) · Twilio (SMS) · PayOS (payments) · `testify` for tests.
This is a REST API consumed by the React SPA in `../frontend-app` over `/api/v1`.

## Build & Test

First run `cp .env.example .env` and fill in secrets (`openssl rand -base64 64` for `JWT_SECRET`).

```bash
make up               # start postgres (:5432) + redis (:6379) via docker-compose
make migrate-up       # apply all db/migrations
make run              # run API → http://localhost:8080/healthz  (go run ./cmd/api)
make build            # build binary → bin/api

make test-unit        # go test ./... -race  (no DB)
make test-integration # spins up wearwhere_test DB, runs -tags=integration
make test             # unit + integration
make migrate-create name=add_xxx   # new migration pair
```

PayOS defaults to `PAYOS_MODE=mock` (no creds; dev checkout at `/dev/payos/mock-checkout`).
If `SMTP_HOST` / `TWILIO_ACCOUNT_SID` are empty, email/SMS OTPs print to **stdout**.

## Architecture Overview

Modular monolith. **All dependency wiring lives in [`cmd/api/main.go`](cmd/api/main.go)** —
read it first to see how a feature is assembled. Each business domain is a vertical slice
under `internal/<domain>/` with the same four layers:

```
handler/   Gin handlers + routes.go(Mount func) — binds JSON, no business logic
service/   business logic; returns typed *httpx.AppError on failure
repo/      persistence: pgx queries (XxxPG) / Redis stores (XxxRedis)
domain/    entities, DTOs, and AppError vars (errors.go) — no framework imports
middleware/ per-domain guards (e.g. brand/middleware.BrandContext)
```

Domains: `auth`, `brand`, `product`, `cart`, `wishlist`, `customeraddr`, `order`,
`payment` (PayOS), `shipping`, plus background `jobs/` (reservation_cleanup).
Cross-cutting code: `internal/shared/` (postgres, redis, jwt, hash, mailer, sms, storage,
cors, apple, slug, validator) and `pkg/httpx/` (response envelope + `AppError`).
DB schema is owned by `db/migrations/` (numbered up/down pairs) — never hand-edit tables.

Routing (`main.go`): `/api/v1` group has `OptionalAuth` + Redis `RateLimit`; role-gated
subgroups `/brand/me` (`RequireAuth`+`RequireRole(brand)`+`BrandContext`) and `/me`
(`RequireAuth`+`RequireRole(customer)`). Graceful shutdown on SIGINT/SIGTERM.

## Conventions & Patterns

- **Layer discipline**: handler → service → repo. A handler must never call a repo directly.
- **Errors**: declare typed errors as `httpx.NewAppError(status, CODE, msg)` vars in
  `domain/errors.go`; services return them; handlers translate via `httpx.ErrorFromApp`.
  Responses always use the envelope helpers in `pkg/httpx` (`OK`/`Created`/`NoContent`/
  `Error`); error JSON is `{ "error": { "code", "message", "details?" } }`.
- **New endpoint**: add to `internal/<domain>/handler/routes.go` (`Mount`), implement
  handler → service → repo, then register the deps in `cmd/api/main.go`. Mirror an
  existing domain (e.g. `cart` or `wishlist`) rather than inventing a new shape.
- **Naming**: one package per layer; repos `XxxPG`/`XxxRedis`; constructors `NewXxx(...)`.
- **Config**: add env keys to `.env.example` and load them in `internal/config/config.go`.
- **Tests**: colocate `*_test.go`; DB-backed tests use the `//go:build integration` tag.
- Passwords: bcrypt cost 12. Refresh tokens: opaque, SHA-256-hashed in DB, rotated on refresh.
